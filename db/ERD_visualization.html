  • DEVELOPERS ‚Üí MODELS: One-to-Many (1:*) - One developer can create multiple models
  • USERS ‚Üí USER_HISTORY: One-to-Many (1:*) - One user can have multiple interaction records
  • MODELS ‚Üí USER_HISTORY: One-to-Many (1:*) - One model can be used in multiple interactions
  • USERS ‚Üí MODEL_REVIEWS: One-to-Many (1:*) - One user can write multiple reviews
  • MODELS ‚Üí MODEL_REVIEWS: One-to-Many (1:*) - One model can receive multiple reviews
  • Multiple entities ‚Üí API_USAGE_LOGS: For comprehensive API usage tracking

üìã Business Rules

  • Each user can only review a specific model once (unique constraint)
  • Models must have a developer owner (cannot exist without developer)
  • Cascade deletes maintain referential integrity
  • Status management controls access for both users and developers
  • API rate limiting through developer quotas
  • Automatic calculation of statistics through database triggers
